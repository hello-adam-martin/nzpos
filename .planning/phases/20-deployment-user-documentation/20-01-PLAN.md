**Section 1: Supabase Production Setup (DEPLOY-01)**
Per D-01, this is the first section. Include:
1. Create new Supabase project at supabase.com (Organization > New Project)
2. Region: Australia (Sydney) ap-southeast-2 — closest to NZ
3. Note project ref for CLI linking
4. Retrieve from Settings > API: Project URL (`NEXT_PUBLIC_SUPABASE_URL`), anon key (`NEXT_PUBLIC_SUPABASE_ANON_KEY`), service_role key (`SUPABASE_SERVICE_ROLE_KEY`)
5. Apply migrations — list all 23 migrations (001 through 023) in a table with filename and description. Application command:
   ```bash
   npx supabase link --project-ref <your-project-ref>
   npx supabase db push
   npx supabase migration list  # verify all applied
   ```
6. Auth configuration in dashboard:
   - Authentication > URL Configuration > Site URL: `https://yourdomain.com`
   - Authentication > URL Configuration > Redirect URLs: `https://*.yourdomain.com/**`
   - Authentication > Hooks > Custom Access Token: verify the hook from migration 003 appears
7. Storage buckets — verify `product-images` (from migration 004) and `store-logos` (from migration 018) exist in Storage > Buckets
8. Super admin seed — two-step process:
   - Create user via Authentication > Users > Add User (email + password)
   - Run SQL in SQL Editor:
     ```sql
     UPDATE auth.users
     SET raw_app_meta_data = raw_app_meta_data || '{"is_super_admin": true}'::jsonb
     WHERE email = 'your-super-admin@email.com';
     ```
   - Note: seed.ts is dev-only. Production starts clean.

> Important callout: JWT claims read from `raw_app_meta_data` not `user_metadata` — the super admin flag MUST be in `raw_app_meta_data`.

**Section 2: Stripe Live Key Configuration (DEPLOY-02)**

   - Copy each `price_...` ID for env vars
4. Register Webhook Endpoint 1 (order payments):
   - URL: `https://yourdomain.com/api/webhooks/stripe`
   - Events: read from the route.ts file to get exact event list
   - Save signing secret (`whsec_...`) -> `STRIPE_WEBHOOK_SECRET`
5. Register Webhook Endpoint 2 (billing subscriptions) — SEPARATE endpoint:
   - URL: `https://yourdomain.com/api/webhooks/stripe/billing`
   - Events: read from the billing route.ts to get exact event list
   - Save signing secret -> `STRIPE_BILLING_WEBHOOK_SECRET`

> Warning callout: These two endpoints use DIFFERENT signing secrets. Using the same secret for both causes `constructEvent()` to throw 400 on every billing webhook request.

6. Webhook replay verification: after deploy, send a test event from Stripe dashboard and verify 200 in Vercel function logs

**Section 3: Vercel Production Deploy with Wildcard DNS (DEPLOY-03)**
Per D-04, step-by-step DNS guide with NS delegation:
1. Import project to Vercel (link GitHub repo)
2. Add root domain in Project > Settings > Domains (e.g., `nzpos.co.nz`)
3. Vercel displays NS records (e.g., `ns1.vercel-dns.com`, `ns2.vercel-dns.com`)
4. At domain registrar: change nameservers to Vercel's NS records

> Warning callout per D-04: Wildcard SSL requires NS record delegation, NOT a CNAME record. A CNAME for `*.yourdomain.com` will NOT provision wildcard SSL. You must delegate nameservers to Vercel.

5. Propagation: 24-48 hours. Check with `dig NS yourdomain.com`
6. Once propagated, Vercel auto-provisions wildcard SSL for `*.yourdomain.com`
7. Verify: visit `https://test.yourdomain.com` — SSL cert should show `*.yourdomain.com`

8. Environment variables in Vercel (Project > Settings > Environment Variables):
   - List all 24 vars from .env.example, with production-specific values called out:
     - `NEXT_PUBLIC_SUPABASE_URL` -> production Supabase URL
     - `ROOT_DOMAIN` and `NEXT_PUBLIC_ROOT_DOMAIN` -> `yourdomain.com` (NOT `lvh.me:3000`)
     - `NEXT_PUBLIC_BASE_URL` and `NEXT_PUBLIC_SITE_URL` -> `https://yourdomain.com`
     - `XERO_REDIRECT_URI` -> `https://yourdomain.com/api/xero/callback`
     - Both Stripe keys -> live keys (not `sk_test_...`)
   - Cross-reference: see docs/env-vars.md for full descriptions

9. Middleware verification post-deploy:
   - Visit root domain -> marketing page loads
   - Visit `/super-admin` -> redirect to login
   - Sign up merchant -> subdomain routes correctly
   - Verify `x-store-id` header on admin routes via browser devtools

10. Cron jobs (vercel.json already configured):
    - List the 3 cron jobs with their schedules (read from vercel.json)
    - Verify `CRON_SECRET` is set — requests without it return 401

11. Resend email setup (prerequisite for email notifications):
    - Create account at resend.com
    - Verify domain via DNS TXT record
    - Create API key -> `RESEND_API_KEY`
    - Set `RESEND_FROM_ADDRESS` to verified address

12. CSP note: CSP is currently `Content-Security-Policy-Report-Only` in middleware.ts. Monitor browser console for violations after go-live. When stable, switch header name to `Content-Security-Policy` (requires middleware.ts edit — separate task, not part of deploy).

## Post-Deploy Smoke Test

- [ ] **Email notification**: After step 4, order confirmation email arrives at customer email address
