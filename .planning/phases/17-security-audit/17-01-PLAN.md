  <action>
    Per D-01, audit security domains 1-3 in blast-radius order. Per D-05, document findings only — do NOT fix anything.

    **Domain 1: RLS Tenant Isolation (SEC-01)**
    - Read ALL migration files in order. For every table with `store_id`, verify it has RLS enabled and policies for SELECT/INSERT/UPDATE/DELETE using the canonical pattern: `auth.jwt() -> 'app_metadata' ->> 'store_id'`
    - Specifically verify: `xero_connections` and `xero_sync_log` from migration 008 — research shows they lack role checks (any JWT with matching store_id can access, including customers)
    - Verify `refund_items` has no UPDATE/DELETE policies — confirm this is intentional immutability
    - Check `orders_public_read` policy: read the storefront order confirmation route to determine if it uses `lookup_token` from migration 009 or raw order UUID (potential IDOR)
    - Verify super admin read policies exist on all tenant tables (migration 020)
    - Check that NO policy uses `user_metadata` or `current_setting()` — these are deprecated patterns
    - For tables intentionally without RLS (rate_limits, super_admins), document why

    **Domain 2: Storage Bucket Policies (SEC-02)**
    - Read migration 004 for `product-images` bucket policies
    - Grep for all `supabase.storage.from('product-images').upload(` calls to determine if uploads use `{store_id}/{filename}` path format
    - Also check `supabase.storage.from('store-assets')` or any other bucket
    - Document whether cross-tenant writes are possible (any authenticated user writing to any store's folder)

    **Domain 3: Auth Flow Verification (SEC-04, SEC-05, SEC-06, SEC-07)**
    - Verify owner auth: middleware calls `getUser()` (server-side JWT validation, not just decode), session cookies are httpOnly
    - Verify staff PIN: `staffPin.ts` has lockout at 10 attempts / 5-min window, JWT expires in 8h, middleware validates via `jwtVerify`
    - Verify super admin: middleware checks `app_metadata.is_super_admin === true` for `/super-admin` routes. Read ALL files in `src/actions/super-admin/` and verify each has action-level `user.app_metadata?.is_super_admin !== true` guard
    - Verify customer isolation: middleware blocks customer role from `/pos` and `/admin` paths. Check whether any Server Action in `src/actions/` could be called by a customer JWT that should be owner/staff-only
    - Verify JWT claims hook (migration 016): confirm claims are written to `app_metadata` (not `user_metadata`), order is super_admin → staff → customer

    **Domain 4: SECURITY DEFINER RPCs (SEC-03)**
    - Find all `SECURITY DEFINER` functions across migrations
    - For each: verify GRANT/REVOKE restricts who can EXECUTE (should be `service_role` only or specific roles)
    - Check `increment_promo_uses` and `restore_stock` — research flags they accept arbitrary UUIDs with no caller validation
    - Do NOT add `auth.uid()` checks inside SECURITY DEFINER functions (auth context is the DB owner, not the caller)

    **Domain 5: Webhook Integrity (SEC-11)**
    - Verify both webhook route handlers call `stripe.webhooks.constructEvent()` with the correct env var
    - Confirm order webhook uses `STRIPE_WEBHOOK_SECRET`, billing uses `STRIPE_BILLING_WEBHOOK_SECRET`
    - Verify raw body is read correctly (not parsed JSON)
    - Check error handling: does a signature failure return 400, not 200?

    Write findings for domains 1-5 to the SECURITY-AUDIT.md file in this format:
    ```
    ## Domain N: [Name]
    ### Requirement: SEC-XX
    ### Findings
    - **F-{N}.{M}**: [Finding description]
      - Severity: Critical | High | Low
      - File: [path]
      - Evidence: [code snippet or grep result]
      - Recommendation: [what to fix]
    ### Verdict: [PASS | PASS WITH FINDINGS | FAIL]
    ```

    **Domain 6: Server Action Validation (SEC-08)**
    - Read ALL 48 action files in `src/actions/` (use `find src/actions -name "*.ts" -not -path "*__tests__*" -not -path "*.test.*"`)
    - For each action file, check:
      1. Does it have `'use server'` directive?
      2. Does it import and use `z` from `zod` with `.safeParse()` or `.parse()` before any DB operation?
      3. Does it return raw `dbError.message` or `error.message` to the client? (stack trace leak risk)
    - Classify:
      - Actions that take user-controlled input but lack Zod: **High** (createCheckoutSession, importProducts, checkSlugAvailability, validatePromoCode)
      - Actions that take no external input and lack Zod: **Low** (signOut, customerSignOut, dismissWizard, disconnectXero, triggerManualSync, createBillingPortalSession)
      - Actions that leak raw DB errors: **High** (createProduct, updateProduct, deactivateProduct, closeCashSession, openCashSession, importProducts)
    - Create a complete table of all 48 actions with columns: File | Has Zod | Has server-only | Leaks Errors | Severity

    **Domain 7: Secrets & Environment Hygiene (SEC-09, SEC-10)**

    - Cross-check against `.env.example` — list every var missing from .env.example
    - Per D-10, scan current HEAD for hardcoded secrets: grep for patterns like API keys, passwords, tokens in source (not .env files)
    - Check `server-only` status:
      - Files that import `createSupabaseAdminClient` (uses service_role key directly): **High** if missing server-only
      - Files that import `createSupabaseServerClient` only: **Low** if missing server-only (transitive guard exists)
    - List all 23 action files missing `server-only` with their classification (Group B1 High vs Group B2 Low from research)

    **Domain 8: Security Headers (SEC-12)**
