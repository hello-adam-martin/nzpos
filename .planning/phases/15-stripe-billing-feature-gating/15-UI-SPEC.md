## Interaction Contract

### Checkout Flow (Subscribe / Start trial)

1. Merchant clicks "Start free trial" or "Subscribe" on an add-on card.
2. Button shows loading state: spinner replaces text, button disabled. 150ms transition.
3. Server Action `createSubscriptionCheckoutSession(feature)` fires.
4. On success: `router.push(checkoutUrl)` — redirects to Stripe Checkout (hosted page).
5. On error: button re-enables, inline error message appears below button (14px `--color-error`).
6. On return from Stripe (success URL): page reloads, webhook already fired, add-on card renders in Active state.
7. On return from Stripe (cancel URL): page reloads at billing page, add-on still in Inactive state. No error shown.

### Billing Portal Flow (Manage)

1. Merchant clicks "Open billing portal".
2. Button shows loading state: spinner, disabled. 150ms.
3. Server Action `createBillingPortalSession()` fires.
4. On success: `router.push(portalUrl)`.
5. On error: inline error below button (14px `--color-error`).

### Upgrade Prompt

Inline in the gated feature area. No interaction beyond clicking the amber CTA which navigates to `/admin/billing?upgrade={feature}`. The billing page scrolls to or highlights the matching add-on card when the `upgrade` query param is present.

### Loading States

All Server Action buttons: replace label text with an inline spinner (16px, same color as button text), disable button. Duration: until response. No skeleton loaders on page load — Server Component renders synchronously.

### Admin Sidebar — Billing Nav Link

Add "Billing" link to `AdminSidebar` between "Settings" and the bottom of the nav. Uses the same nav item pattern as existing links. No badge or notification count needed.

## Phase-Specific Notes

1. **Billing page is admin-only.** Storefront and POS surfaces have no billing UI. All billing interaction is at `/admin/billing`.

2. **Upgrade prompts live at the gated feature location.** They do not pop up as modals or toasts. Merchants see the prompt in context — where the feature would be used.

3. **No custom cancel UI.** The Stripe Customer Portal handles all subscription management. The only cancel path is through the portal link. This is by design (CONTEXT.md D-06).

4. **`stripe_customer_id` is created on first checkout.** Until a merchant clicks Subscribe or Start free trial for the first time, there is no Stripe Customer. The "Open billing portal" button should be hidden or disabled until a `stripe_customer_id` exists on the store record. Show: "Subscribe to an add-on to access the billing portal."

5. **Price values are runtime data.** Do not hardcode NZD prices in copy. Fetch from Stripe Products API or a config table seeded from env vars. Render as `{formatNZD(priceCents)}/month` using the existing `formatNZD` utility.

6. **`?upgrade={feature}` query param.** When billing page is loaded with this param, scroll the matching add-on card into view and apply a brief highlight ring (`ring-2 ring-[var(--color-amber)] ring-offset-2`) for 2 seconds, then fade the ring off. This visual affordance orients the merchant who arrived via an upgrade prompt.
