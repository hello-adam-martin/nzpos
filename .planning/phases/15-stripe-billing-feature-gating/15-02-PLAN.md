    Create `src/app/api/webhooks/stripe/billing/route.ts` — a separate Route Handler for subscription events. Per D-05, handles `customer.subscription.created`, `customer.subscription.updated`, `customer.subscription.deleted`.

    Implementation (follow Research Pattern 3 exactly):

    1. **Signature verification** using `STRIPE_BILLING_WEBHOOK_SECRET` env var (NOT the existing STRIPE_WEBHOOK_SECRET)
    2. **Event routing:** Only handle the three subscription event types. Return 200 for unrecognized events.
    3. **handleSubscriptionChange function:**
       - Idempotency check via `stripe_events` table (same pattern as existing webhook)
       - Extract `customerId` from `subscription.customer` (handle string or object)
       - Resolve `store_id`: first try `subscription.metadata.store_id`, then fallback to `stores` table lookup by `stripe_customer_id` (per Research Pitfall 1)
       - If no store_id found, log error and return (don't crash)
       - Upsert `stores.stripe_customer_id` — `UPDATE stores SET stripe_customer_id = customerId WHERE id = storeId AND stripe_customer_id IS NULL` (captures on first event)
       - Map `subscription.items.data[0].price.id` to feature column via `PRICE_TO_FEATURE` from `src/config/addons.ts`
       - Determine `isActive`: true when event is NOT `customer.subscription.deleted` AND status is `active` or `trialing`
       - Update `store_plans` SET `[featureColumn] = isActive, updated_at = new Date().toISOString()` WHERE `store_id = storeId`
       - Record event in `stripe_events` table for idempotency

    Import `PRICE_TO_FEATURE` from `@/config/addons`.
    Import `createSupabaseAdminClient` from `@/lib/supabase/admin`.
    Create a new Stripe instance directly (do NOT import from `@/lib/stripe` — webhook handlers need raw constructor for `constructEvent`).

    Do NOT use `import 'server-only'` — Route Handlers already run server-side and this import can cause build issues in API routes.
  </action>
