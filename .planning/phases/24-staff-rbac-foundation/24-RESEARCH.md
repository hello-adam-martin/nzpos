# Phase 24: Staff RBAC Foundation - Research

**Researched:** 2026-04-05
**Domain:** Role-Based Access Control, Staff Management, Next.js Middleware, JWT sessions
**Confidence:** HIGH

## Summary

This phase adds a Manager role tier and staff management UI to an existing system that already has Owner and Staff roles. The core codebase is mature — 26 migrations, 48 Server Actions with consistent patterns, and an established JWT session architecture for staff PIN logins. The work is additive: extend the DB constraint, add a centralized roles constant, update middleware for manager admin access, build the staff management page and Server Actions, and update AdminSidebar for role-aware filtering.

The key architectural challenge is the two-session-system design: owners authenticate via Supabase Auth (app_metadata.role), while staff authenticate via a custom jose JWT cookie (staff_session). The middleware currently gates all /admin/* routes to Supabase Auth owners only (lines 115-181 of middleware.ts). Managers must reach /admin/* via their staff JWT — requiring a surgical middleware update that checks the staff_session cookie for manager role when no Supabase Auth owner session is present.

The session invalidation mechanism (pin_locked_until = now()) is already decided and partially implemented — the staffPin.ts action already checks pin_locked_until at login. Role changes and deactivation simply write this field to force re-login, no new infrastructure needed.

**Primary recommendation:** Work in strict dependency order: DB migration → roles constant → resolveStaffAuthVerified() → middleware update → Server Actions → Admin UI → Sidebar filtering. Each step gates the next.

### Pattern 2: resolveStaffAuthVerified() — DB Role Lookup for Writes

**What:** New function in resolveAuth.ts that verifies the staff JWT AND does a live DB lookup to confirm the current role. Never trusts the JWT role for authorization of writes.
**When to use:** All new staff CRUD Server Actions that must enforce role restrictions.

```typescript
// src/lib/resolveAuth.ts — EXTEND with:
// Source: STATE.md decision — "resolveStaffAuthVerified() does DB role lookup on all role-gated mutations"

export async function resolveStaffAuthVerified(): Promise<{
  store_id: string
  staff_id: string
  role: PosRole
} | null> {
  // 1. Verify JWT is valid (same as resolveStaffAuth)
  const staffAuth = await resolveStaffAuth()
  if (!staffAuth) return null

  // 2. DB lookup to get CURRENT role (JWT may be stale after role change)
  const supabase = createSupabaseAdminClient()
  const { data: staff } = await supabase
    .from('staff')
    .select('role, is_active')
    .eq('id', staffAuth.staff_id)
    .eq('store_id', staffAuth.store_id)
    .single()

  if (!staff || !staff.is_active) return null

  return {
    store_id: staffAuth.store_id,
    staff_id: staffAuth.staff_id,
    role: staff.role as PosRole,
  }
}
```

### Pattern 3: Server Action Role Guard (D-08)

**What:** All staff management actions require owner role. processRefund/processPartialRefund allow manager. Pattern: check auth → check role → typed error.
**When to use:** Every restricted Server Action in this phase.

### Pattern 4: Middleware Manager Admin Access (D-02, D-03)

**What:** The /admin/* block in middleware currently requires Supabase Auth owner session. Must be extended to also allow staff JWT with role='manager', with route-level filtering for manager-only routes.
**When to use:** middleware.ts — the /admin block starting at line 115.

The key insight: middleware cannot call resolveStaffAuth() directly (it uses cookies() from next/headers which isn't available in middleware). Middleware must parse the JWT directly using jwtVerify — which is already done in the /pos block (lines 200-213).

### Pattern 6: PIN Generation and Validation

**What:** Auto-generate cryptographically random 4-digit PINs. Validate against blacklist and uniqueness.
**When to use:** createStaff and resetStaffPin actions.

```typescript
// src/lib/pin.ts — NEW utility
import crypto from 'crypto'

const PIN_BLACKLIST = ['0000', '1111', '2222', '3333', '4444', '5555', '6666', '7777', '8888', '9999', '1234', '4321']

export function generatePin(): string {
  let pin: string
  do {
    // crypto.randomInt(0, 10000) gives [0, 9999]
    pin = crypto.randomInt(0, 10000).toString().padStart(4, '0')
  } while (PIN_BLACKLIST.includes(pin))
  return pin
}

export function isPinBlacklisted(pin: string): boolean {
  return PIN_BLACKLIST.includes(pin)
}
```

Duplicate PIN check: query `staff` table for existing `pin_hash` values is impractical (hashes are not comparable). Instead: generate PIN → attempt bcrypt compare against each active staff PIN hash. For stores with ≤20 staff this is fast. Alternatively, store a plaintext PIN index (not recommended). The pragmatic approach: generate, check against blacklist only, accept low probability of collision (4-digit space = 10,000 possibilities minus blacklist).

**Simpler approach used in practice:** Accept the birthday problem risk at small team sizes. With ≤10 staff and 9,988 non-blacklisted PINs, collision probability is <0.5%. Document this as acceptable for v1.

### Pattern 7: Database Migration for Manager Role

**What:** Alter the staff table CHECK constraint to add 'manager'. Use a standard numbered migration file.
**When to use:** Wave 0 — must precede all other work.

```sql
-- supabase/migrations/027_staff_manager_role.sql
-- Add 'manager' to the staff role CHECK constraint

ALTER TABLE public.staff
  DROP CONSTRAINT staff_role_check;

ALTER TABLE public.staff
  ADD CONSTRAINT staff_role_check
  CHECK (role IN ('owner', 'manager', 'staff'));
```

Note: PostgreSQL requires DROP + re-ADD for CHECK constraints (no ALTER for check constraint values). The constraint name is `staff_role_check` based on the default naming convention from migration 001 (`role TEXT NOT NULL CHECK (role IN ('owner', 'staff'))`). Verify actual constraint name with `\d staff` or Supabase dashboard before writing migration.

### Anti-Patterns to Avoid

- **Trusting JWT role for writes:** The JWT role can be stale after a role change. Always use resolveStaffAuthVerified() (DB lookup) for any Server Action that modifies data or enforces access control.
- **Using server client (RLS) for staff mutations:** The staff table RLS is owner-scoped (Supabase Auth). Staff management actions must use `createSupabaseAdminClient()` for queries against the staff table.
- **Hiding UI without server-side enforcement:** D-08 is explicit — UI hiding is the primary UX guard, Server Action role check is defense-in-depth. Both must be implemented.
- **Checking email on manager layout:** Managers have no Supabase Auth session and therefore no `user.email`. The layout must handle this and display staff name instead.
- **Processing manager routes through setup wizard redirect:** Lines 164-175 in middleware.ts redirect to /admin/setup if setup_wizard_dismissed is false. This logic must not apply to manager sessions — managers skip the setup wizard check.

## Common Pitfalls

### Pitfall 1: Middleware Manager Check Must Come After Owner Check Fails

**What goes wrong:** If manager JWT check runs before the Supabase Auth check, a valid owner session could be bypassed and the owner receives a manager-filtered view.
**Why it happens:** Code ordering mistake — owner check and manager check are in the wrong sequence.
**How to avoid:** Structure the /admin block as: (1) try Supabase Auth → if owner pass through, (2) try staff JWT → if manager validate route, (3) fall through to login. Never check staff JWT if owner session was found.
**Warning signs:** Owner seeing manager-restricted sidebar instead of full sidebar.

### Pitfall 2: Admin Layout Reads Supabase Auth User — Managers Have None

**What goes wrong:** `AdminLayout` calls `supabase.auth.getUser()` and uses `user?.app_metadata?.role` to determine display state. For managers, this returns null — the layout breaks or falls back to wrong defaults.
**Why it happens:** The layout was designed for owner-only. Manager access is new.
**How to avoid:** In `AdminLayout`, after the Supabase Auth check fails, also check the `staff_session` cookie using `jwtVerify`. Extract role and staff_id. Pass both `role` and `staffName` (from DB lookup or JWT) to AdminSidebar.
**Warning signs:** Manager sees owner sidebar, or sidebar crashes on null user.

### Pitfall 3: Setup Wizard Redirect Blocks Manager Access

**What goes wrong:** Lines 162-175 of middleware.ts redirect any /admin/* request to /admin/setup if `setup_wizard_dismissed` is false. This check fires BEFORE reaching the manager staff JWT check in the current refactor.
**Why it happens:** The setup wizard check is inside the owner auth block but the logic ordering affects manager flow.
**How to avoid:** The setup wizard redirect only applies when the user is an authenticated owner. When a manager JWT is validated, skip the setup wizard DB check entirely.
**Warning signs:** Managers get redirected to /admin/setup which they have no access to.

### Pitfall 4: processRefund and processPartialRefund Only Check Owner Session

**What goes wrong:** Both refund actions check `supabase.auth.getUser()` for owner auth only (line 22-24 of processRefund.ts). Managers cannot call these even though D-06/STAFF-06 says they should be able to process refunds.
**Why it happens:** The actions predate the manager role concept.
**How to avoid:** Update both processRefund and processPartialRefund to fall back to resolveStaffAuthVerified() when no owner session is found, and allow 'manager' role to proceed.
**Warning signs:** Manager clicks refund button, gets "Not authenticated" error even though UI is visible.

### Pitfall 5: PIN Column Constraint Check Name

**What goes wrong:** Migration 027 drops `staff_role_check` but the actual constraint name in production might differ (e.g., PostgreSQL auto-names inline CHECK constraints as `staff_role_check1` or similar).
**Why it happens:** Inline CHECK constraints in CREATE TABLE get auto-named by PostgreSQL in an implementation-specific way.
**How to avoid:** In migration 027, use `DROP CONSTRAINT IF EXISTS` for both possible names, or use a more defensive approach: `ALTER TABLE staff DROP CONSTRAINT staff_role_check;` preceded by a comment noting to verify the name. The implementor should run `SELECT constraint_name FROM information_schema.table_constraints WHERE table_name = 'staff' AND constraint_type = 'CHECK';` to confirm the name.
**Warning signs:** Migration fails with "constraint not found".

### Pitfall 6: resolveStaffAuthVerified Hits DB on Every Admin Page Load

**What goes wrong:** If AdminLayout calls resolveStaffAuthVerified() on every render for manager sessions, it adds a DB round-trip to every admin page load.
**Why it happens:** Correct security practice (fresh DB lookup) conflicts with performance.
**How to avoid:** In AdminLayout, use the JWT role (from jwtVerify) for display-only decisions (sidebar filtering, name display). Only call resolveStaffAuthVerified() in Server Actions that mutate data. The middleware already validated the JWT — the layout can trust it for rendering.
**Warning signs:** Admin pages slow for managers; DB query count spikes.

### Pitfall 7: Deactivated Staff Can Still Use Existing Valid JWT

**What goes wrong:** Deactivating a staff member sets `is_active = false` and `pin_locked_until = now()`. But an existing valid staff_session JWT cookie remains in their browser. If middleware only checks JWT validity (not DB is_active), the session persists until JWT expires (8 hours).
**Why it happens:** Stateless JWT by design — but deactivation requires immediate invalidation.
**How to avoid:** Middleware's /pos and /admin manager checks only validate JWT cryptography. For deactivated staff, the real enforcement is in verifyStaffPin (which checks is_active at login) and in resolveStaffAuthVerified (which checks is_active for writes). For immediate POS enforcement, add a DB is_active check in the middleware POS block — but this adds a DB call per request. Acceptable tradeoff: deactivated staff lose access at next action (Server Action) or next page load if middleware checks DB. Document this as the decided behavior from D-12.

**Clarification from D-12:** pin_locked_until = now() blocks the NEXT login attempt. For a currently-logged-in deactivated staff member, their existing JWT remains valid until expiry (up to 8 hours). This is the accepted tradeoff for stateless sessions. The implementor should add an is_active check in resolveStaffAuthVerified to refuse writes from deactivated staff.

### createStaff Server Action

```typescript
// src/actions/staff/createStaff.ts
'use server'
import 'server-only'
import { revalidatePath } from 'next/cache'
import { createSupabaseServerClient } from '@/lib/supabase/server'
import { createSupabaseAdminClient } from '@/lib/supabase/admin'
import { CreateStaffSchema } from '@/schemas/staff'
import { POS_ROLES } from '@/config/roles'
import bcryptjs from 'bcryptjs'

export async function createStaff(input: unknown) {
  // 1. Auth — owner only (staff management is owner-only per D-07)
  const supabase = await createSupabaseServerClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return { error: 'Not authenticated' }
  const storeId = user.app_metadata?.store_id as string | undefined
  if (!storeId) return { error: 'No store context' }
  if (user.app_metadata?.role !== POS_ROLES.OWNER) return { error: 'INSUFFICIENT_ROLE' }

  // 2. Validate input
  const parsed = CreateStaffSchema.safeParse(input)
  if (!parsed.success) return { error: parsed.error.flatten().fieldErrors }
  const { name, pin, role } = parsed.data

  // 3. Hash PIN
  const pin_hash = await bcryptjs.hash(pin, 10)

  // 4. Insert — admin client bypasses RLS (staff table RLS is owner-scoped via Supabase Auth)
  const adminClient = createSupabaseAdminClient()
  const { data: staff, error: dbError } = await adminClient
    .from('staff')
    .insert({ store_id: storeId, name, pin_hash, role, is_active: true })
    .select('id')
    .single()

  if (dbError) {
    console.error('[createStaff] DB error:', dbError)
    return { error: { _form: ['Failed to create staff member'] } }
  }

  revalidatePath('/admin/staff')
  return { success: true, staffId: staff.id }
}
```

```typescript
if (parsed.data.name) updateData.name = parsed.data.name
if (parsed.data.role) {
  updateData.role = parsed.data.role
  // D-12: Invalidate current session — forces re-login
  updateData.pin_locked_until = new Date().toISOString()
}

const { error: dbError } = await adminClient
  .from('staff')
  .update(updateData)
  .eq('id', parsed.data.staffId)
  .eq('store_id', storeId) // tenant isolation

if (dbError) return { error: 'Failed to update staff member' }

revalidatePath('/admin/staff')
return { success: true }
```
