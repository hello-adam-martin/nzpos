# Phase 24: Staff RBAC Foundation - Context

**Gathered:** 2026-04-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Store owners can manage their staff roster (add, edit, deactivate, reset PIN) and the system enforces role-based permissions across three tiers: Owner (full admin), Manager (filtered admin + POS), Staff (POS only). Manager role is new — currently only owner and staff exist in the database.

</domain>

<decisions>
## Implementation Decisions

### Manager Admin Access
- **D-01:** Managers get a filtered admin panel — they can see dashboard, orders, reports, and process refunds. Products, staff, settings, and billing sidebar links are hidden.
- **D-02:** Managers access admin via a "Go to Admin" button on the POS screen after PIN login (same pattern as owners). No separate auth system — PIN login is sufficient.
- **D-03:** If a manager navigates directly to a restricted admin route (e.g. /admin/products via URL), silently redirect to /admin/dashboard. No error page.

### Staff List UI & Actions
- **D-04:** Staff management page uses a table with inline actions (Edit, Deactivate, Reset PIN), consistent with existing products/orders table patterns in admin.
- **D-05:** Adding a new staff member uses a modal form with name + role fields. PIN is auto-generated by default, with a manual override toggle for owners who want to set a specific PIN. Basic validation required (no 0000, no duplicate PINs within the store).
- **D-06:** PIN display (on create or reset) uses a success modal with the 4-digit PIN in large text, a "Copy" button, and a warning that it won't be shown again. Must dismiss to continue.

### Permission Enforcement
- **D-07:** Three-tier role model: Owner (full admin), Manager (filtered admin + POS), Staff (POS only). Staff role has zero admin access — POS only, matching current behavior.
- **D-08:** Restricted Server Actions throw a typed unauthorized error (e.g. 'INSUFFICIENT_ROLE') when called by an unauthorized role. UI hides restricted actions as primary guard; server-side check is defense-in-depth.
- **D-09:** Manager admin sidebar shows only: Dashboard, Orders, Reports, Cash-up. All other links (Products, Staff, Settings, Billing, Integrations, Inventory, Promos) are hidden.

### Role Change UX
- **D-10:** Role changes require a confirmation modal: "Change [Name] from [Current] to [New]? They will be logged out and need to re-enter their PIN." Confirm/Cancel.
- **D-11:** Deactivation requires a separate confirmation modal: "Deactivate [Name]? They will be immediately logged out and unable to use the POS." Prominent warning.
- **D-12:** On role change or deactivation, system sets `pin_locked_until = now()` to invalidate the staff member's current JWT session, forcing re-login (or blocking login if deactivated). Already decided in research.

### Claude's Discretion
- Exact table column widths and responsive breakpoints for the staff table
- Loading skeleton patterns for the staff list
- Error toast styling for failed operations
- Exact wording of confirmation modals (general tone: clear, direct, NZ-friendly)

</decisions>

### Established Patterns
- **Server Actions with Zod validation**: All 48 existing Server Actions use Zod input validation + server-only guard.
- **resolveAuth() → { store_id, staff_id }**: Standard auth resolution returning snake_case fields.
- **SECURITY DEFINER RPCs**: Atomic DB operations for sensitive mutations (stock, provisioning). Staff role changes should follow this pattern.
- **Feature gating via JWT claims**: hasXero, hasEmailNotifications, hasInventory in JWT. Role already in JWT but currently only 'owner'/'staff'.

- **Database CHECK constraint**: ALTER to add 'manager' to allowed role values.
- **AdminSidebar**: Role-aware nav filtering.
