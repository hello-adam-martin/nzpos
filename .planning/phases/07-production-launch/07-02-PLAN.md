Human performs these steps in order:

**Step 1: Create Supabase production project**
1. Go to https://supabase.com/dashboard
2. Click "New Project"
3. Choose organization, name project (e.g., "nzpos-production"), select region (ap-southeast-2 Sydney for NZ latency)
4. Set a strong database password and save it securely
5. Wait for project to provision (1-2 minutes)
6. Note the project ref from the URL: `app.supabase.com/project/<PROJECT_REF>`

**Step 2: Copy production credentials**
From Supabase Dashboard > Project Settings > API, copy:
- `NEXT_PUBLIC_SUPABASE_URL` (the Project URL, e.g., https://abcdefgh.supabase.co)
- `NEXT_PUBLIC_SUPABASE_ANON_KEY` (the anon public key)
- `SUPABASE_SERVICE_ROLE_KEY` (the service_role key — keep secret)

**Step 3: Link Supabase CLI and push migrations**
Run in the project root terminal:
```bash
supabase link --project-ref <PROJECT_REF>
supabase db push
supabase migration list --linked
```
Verify output shows all 9 migrations (001 through 009) as "Applied".

**Step 4: Register auth hook in Supabase Dashboard**
1. Go to Supabase Dashboard > Authentication > Hooks
2. Enable "Custom Access Token" hook
3. Set schema to `public`, function name to `custom_access_token_hook`
4. The URI should be: `pg-functions://postgres/public/custom_access_token_hook`
5. Save

WARNING: If this step is skipped, authenticated users will see EMPTY data everywhere. The auth hook injects store_id into JWTs — without it, RLS blocks all queries silently.

**Step 5: Verify storage bucket exists**
Go to Supabase Dashboard > Storage. Confirm a `product-images` bucket exists (created by migration 004). It should be marked as public.

**Step 6: Create owner auth account**
1. Go to Supabase Dashboard > Authentication > Users
2. Click "Invite user" or "Add user" > "Create new user"
3. Enter the owner's email and a password
4. Note the user's UUID from the users list (you will need it for Step 7)

**Step 7: Create store and staff records**
Go to Supabase Dashboard > SQL Editor and run:

```sql
-- Replace values below with real data
-- Get the auth user UUID from Step 6

INSERT INTO public.stores (name, owner_auth_id)
VALUES ('Your Store Name', '<auth-user-uuid-from-step-6>')
RETURNING id;

-- Use the store id returned above and the same auth user UUID
INSERT INTO public.staff (store_id, auth_user_id, name, role)
VALUES (
  '<store-id-from-insert-above>',
  '<auth-user-uuid-from-step-6>',
  'Owner Name',
  'owner'
);
```

NOTE: The stores table has columns: id (auto), name (required), owner_auth_id (required, references auth.users), created_at (auto), updated_at (auto). There is no gst_number or currency column.

**Step 8: Generate STAFF_JWT_SECRET**
Run locally:
```bash
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```
Save the output — you will need it for Vercel env vars.

Human performs these steps in order:

**Step 1: Set up Vercel project**
Option A (GitHub integration — preferred):
1. Go to https://vercel.com/new
2. Import the NZPOS GitHub repository
3. Framework Preset: Next.js (auto-detected)
4. Root Directory: `./` (default)
5. DO NOT deploy yet — set env vars first (Step 2)

Option B (CLI):
```bash
npx vercel link
```

**Step 2: Set ALL environment variables BEFORE first build**
In Vercel Dashboard > Project > Settings > Environment Variables, add these for Production environment:

| Variable | Value | Source |
|----------|-------|--------|
| `NEXT_PUBLIC_SUPABASE_URL` | `https://<ref>.supabase.co` | Supabase Dashboard > API > URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | `eyJ...` | Supabase Dashboard > API > anon key |
| `SUPABASE_SERVICE_ROLE_KEY` | `eyJ...` | Supabase Dashboard > API > service_role key |
| `STAFF_JWT_SECRET` | 64-char hex string | Generated in Task 1, Step 8 |
| `STRIPE_SECRET_KEY` | `sk_test_...` | Stripe Dashboard > Developers > API keys |
| `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` | `pk_test_...` | Stripe Dashboard > Developers > API keys |
| `STRIPE_WEBHOOK_SECRET` | `whsec_placeholder` | Temporary — will update in Plan 03 after webhook registration |

CRITICAL: `NEXT_PUBLIC_*` vars are baked into the build at compile time. If missing, the build will produce an app that calls localhost instead of Supabase production. Set ALL vars BEFORE triggering the first build.

NOTE: Set `STRIPE_WEBHOOK_SECRET` to a placeholder value (e.g., `whsec_placeholder`) for the first deploy. The real value comes from Stripe after registering the webhook endpoint — which requires the production URL (chicken-and-egg). Plan 03 will update this and redeploy.

**Step 3: Trigger first production deploy**
Option A (GitHub integration): Push to main branch, or click "Deploy" in Vercel Dashboard
Option B (CLI): `npx vercel --prod`

**Step 4: Verify deployment**
1. Wait for build to complete (check Vercel Dashboard > Deployments)
2. Visit the production URL: `https://<project>.vercel.app`
3. Verify the storefront loads (product grid, even if empty)
4. Verify the Stripe test mode banner is visible at top: "Test mode active. Payments are simulated. No real charges will occur."
5. Visit `https://<project>.vercel.app/admin/login`
6. Log in with the owner credentials from Task 1 Step 6
7. Verify the admin dashboard loads with data (store name visible, stats cards rendered — may show zeros)
8. Verify Stripe test mode banner is visible in admin too

**Step 5: Verify cron jobs**
In Vercel Dashboard > Project > Crons, confirm two crons are listed:
- `/api/cron/xero-sync` at `0 13 * * *`
- `/api/cron/expire-orders` at `0 14 * * *`

If they do not appear, verify vercel.json is in the project root and redeploy.
