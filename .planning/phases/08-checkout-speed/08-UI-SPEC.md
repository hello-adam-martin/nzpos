# Phase 8 — UI Design Contract: Checkout Speed

> Visual and interaction contract for the barcode scanner overlay and enhanced receipt screen.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

---

## Scope

This phase adds two new UI surfaces to the POS:

1. **Barcode Scanner Overlay** — Camera viewfinder sheet that appears when staff tap a "Scan" button. Decodes EAN-13/UPC-A barcodes and adds the matched product to cart. Shows a clear error state if no match is found and focuses the search bar for manual lookup.
2. **Enhanced Screen Receipt** — Replaces / extends the existing `SaleSummaryScreen` to include store name, full line items, GST breakdown, total, and payment method in a layout compatible with a future physical printer path.

No new surfaces exist outside these two. All other POS chrome is unchanged.

## Surface: BarcodeScannerSheet

### Layout

```
┌─────────────────────────────────────┐
│  fixed inset-0 z-50 bg-navy-dark/90 │
│                                     │
│   [Title]  "Point camera at barcode"│
│                                     │
│  ┌──────────────────────────────┐   │
│  │  camera viewfinder           │   │
│  │  (min 280px tall, rounded-lg)│   │
│  └──────────────────────────────┘   │
│                                     │
│   [Status pill]  idle / scanning /  │
│                  match / no-match   │
│                                     │
│   [Cancel button]  ghost, full-width│
└─────────────────────────────────────┘
```

### States

| State | Visual treatment |
|-------|-----------------|
| Idle | Viewfinder border `border-white/30`. Status pill hidden. |
| Scanning | Animated scan line slides top→bottom (CSS animation, 1.2s linear infinite). Status pill: "Scanning…" in navy/10 bg. |
| Match | Brief 150ms green flash on viewfinder border (`border-success`). Sheet auto-closes. Product added to cart. No explicit confirmation needed — speed is the feature. |
| No match | Viewfinder border turns `border-error`. Status pill: error style (see Copywriting). Sheet closes after 1.5s. Search bar focused. |

### Animation rules

- Scan line: `height: 2px`, `bg-amber/60`, slides from `top-0` to `bottom-0` via CSS `@keyframes scanLine`. Duration: 1200ms, `linear`, `infinite`. Respects `prefers-reduced-motion` — set `animation: none` and show static horizontal center line instead.
- Viewfinder flash (match): `border-color` transition `ease-out 100ms`. Green → back to white/30 only if sheet stays open; otherwise irrelevant as it closes.
- Sheet open/close: `animate-[fadeIn_150ms_ease-out]` — matches existing overlay pattern (`EftposConfirmScreen`, `SaleSummaryScreen`).
- **POS rule from DESIGN.md**: nothing in the POS checkout path animates longer than 150ms. The scan line animation is a UI affordance (not checkout path), so 1200ms is acceptable. All state-change transitions (match, no-match) stay at ≤150ms.

### Accessibility

- `role="dialog"` `aria-modal="true"` `aria-label="Barcode scanner"`.
- Focus trap on mount (same pattern as `EftposConfirmScreen`).
- Cancel button must be keyboard-reachable. `autoFocus` on Cancel when sheet opens — allows iPad hardware keyboard ESC escape.
- Camera permission denied: show error state with copy "Camera access denied. Use the search bar to find products." No retry button — staff must resolve in OS settings.

### ReceiptData type

```ts
// src/lib/receipt.ts
export type ReceiptData = {
  orderId: string
  storeName: string
  completedAt: string          // ISO timestamp
  items: ReceiptLineItem[]
  subtotalCents: number
  gstCents: number
  totalCents: number
  paymentMethod: 'eftpos' | 'cash' | 'split'
  cashTenderedCents?: number
  changeDueCents?: number
}

export type ReceiptLineItem = {
  productId: string
  productName: string
  quantity: number
  unitPriceCents: number
  discountCents: number
  lineTotalCents: number
  gstCents: number
}
```

This type is the single source of truth. `ReceiptScreen` renders it. Future `PrintReceipt` server action consumes it. No screen-only fields — every field must survive serialisation to ESC/POS.

### Differences from current SaleSummaryScreen

| Field | Old `SaleSummaryScreen` | New `ReceiptScreen` |
|-------|------------------------|---------------------|
| Store name | Absent | Present in header (heading weight) |
| Order ID | 8-char uppercase slice | 8-char uppercase, Geist Mono font |
| Data model | Props spread from POSClientShell | Accepts `ReceiptData` object (typed) |
| Export path | Screen only | `ReceiptData` serialisable for printer |

## Interaction Contract

### Barcode scan → cart add flow

1. Staff taps "Scan" button in POS top bar.
2. `BarcodeScannerSheet` opens with `fadeIn_150ms`. Camera stream starts.
3. On successful decode (EAN-13 or UPC-A):
   - Match found: dispatch `ADD_PRODUCT`, sheet closes. Cart updates instantly. No confirmation step.
   - No match: sheet shows error state for 1500ms, then closes. `POSClientShell` focuses the search input.
4. Staff can tap "Cancel" at any time to close without adding.

### Scan button placement

Placed in `POSTopBar` to the right of the store name / left of the staff name display. Icon: `ScanBarcode` from lucide-react (or `Barcode` if `ScanBarcode` unavailable in installed lucide version — check at implementation time). Button style: ghost/outline — `border border-border rounded-md px-3 py-2 text-sm font-medium text-text hover:bg-surface`.

### Receipt → new sale flow

After sale completes, `ReceiptScreen` opens (same `fixed inset-0 z-50` overlay). Staff reviews receipt. Taps "New Sale" — `onNewSale` callback fires, cart resets, overlay closes. No auto-close timer on receipt (staff may need to read the change amount).

## Motion Budget (POS)

| Transition | Duration | Easing | Notes |
|-----------|----------|--------|-------|
| Scanner sheet open | 150ms | ease-out | `fadeIn` — matches all other overlays |
| Scanner sheet close | 100ms | ease-in | Snap close — speed is the feature |
| Viewfinder error flash | 100ms | ease-out | `border-color` to `--color-error` |
| Scan line animation | 1200ms | linear | Affordance only, not in checkout path |
| Receipt open | 150ms | ease-out | `fadeIn` — matches existing `SaleSummaryScreen` |
| Receipt close / new sale | 100ms | ease-in | Immediate cart reset |

`prefers-reduced-motion`: scan line animation set to `none`. All other transitions are ≤150ms and acceptable to keep.
