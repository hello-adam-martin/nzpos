    **Audit and fix error handling patterns (QUAL-02, per D-09, D-10, D-11):**

    1. Scan all Server Actions in `src/actions/`:
       ```bash
       grep -rn "catch" src/actions/ --include="*.ts" | grep -v test | grep -v __tests__
       ```

    2. For each Server Action, ensure the return type matches the standard shape (D-09):
       ```typescript
       { success: boolean; error?: string; data?: T }
       ```

       Common violations to fix:
       - Actions that throw instead of returning `{ success: false, error: '...' }`
       - Actions that return raw error messages containing `err.message` (stack trace leak)
       - Actions that return different shapes (e.g., `{ ok: true }` or `{ result: ... }`)
       - Actions missing try/catch around database operations

    3. For each catch block, ensure:
       - Server-side logging: `console.error('[actionName] store_id=%s error:', storeId, err)` (D-11)
       - Client-side response: `{ success: false, error: 'Something went wrong. Please try again.' }` — NEVER expose `err.message` to client
       - If the error is a known business logic error (e.g., "Out of stock"), use a user-friendly message: `{ success: false, error: 'This item is out of stock.' }`

    4. Scan all Route Handlers in `src/app/api/`:
       ```bash
       grep -rn "catch\|NextResponse.json" src/app/api/ --include="*.ts" | grep -v test | grep -v __tests__
       ```

    5. For each Route Handler, ensure responses match (D-10):
       ```typescript
       NextResponse.json({ success: true, data: result }, { status: 200 })
       NextResponse.json({ success: false, error: 'message' }, { status: 4xx/5xx })
       ```

       Exception: Webhook handlers (Stripe) may return different shapes as required by Stripe's webhook protocol. Don't change webhook response format.

    6. Verify no stack traces leak: `grep -rn "err.message\|error.message\|err.stack\|error.stack" src/actions/ src/app/api/ --include="*.ts" | grep -v test | grep -v __tests__ | grep -v console` — this should return empty (all err.message usage should be in console.error only, not in return values).

    7. Run `npm run test` to verify no test regressions.
  </action>
