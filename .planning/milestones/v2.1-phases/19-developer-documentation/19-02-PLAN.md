    **# Architecture Overview**
    Intro: "NZPOS is a multi-tenant SaaS POS and online store. This document covers the key architectural patterns: authentication, tenant isolation, feature gating, and billing."
    Key files table: middleware.ts, resolveAuth.ts, tenantCache.ts, requireFeature.ts, addons.ts with one-line purpose each.

    **## Authentication**
    Intro: "Three authentication systems operate concurrently, plus a super admin overlay."

    **### Owner Auth (Supabase Auth)** (per D-04):
    - Email/password via Supabase Auth, JWT in cookies via @supabase/ssr
    - `app_metadata.role = 'owner'` and `app_metadata.store_id` set by provisionStore
    - Middleware verifies for /admin routes; resolveAuth() checks owner session first
    - Key files: `src/actions/auth/ownerSignin.ts`, `src/lib/resolveAuth.ts`
    - Mermaid sequence diagram showing: Browser -> Middleware -> SupabaseAuth -> getUser() -> role check -> pass/redirect

    **### Staff PIN Auth (Custom JWT)** (per D-04):
    - PIN verified against bcrypt hash in `staff` table
    - Rate limited: 20/5min IP-level + 10/5min account-level via `check_rate_limit` RPC
    - JWT issued via jose SignJWT (HS256, 8h expiry), stored in HttpOnly `staff_session` cookie
    - Middleware verifies via jwtVerify(staffToken, staffSecret)
    - Key files: `src/actions/auth/staffPin.ts`, `src/middleware.ts`
    - Mermaid sequence diagram showing: Staff -> POS -> verifyStaffPin -> check_rate_limit -> bcrypt.compare -> SignJWT -> cookie

    **### Customer Auth (Supabase Auth)** (per D-04):
    - Email/password via Supabase Auth, `app_metadata.role = 'customer'`
    - Blocked from /admin and /pos in middleware (silent redirect to /)
    - RLS limits to storefront-specific tables
    - Key files: `src/actions/auth/customerSignin.ts`

    **### Super Admin (Supabase Auth + metadata flag)** (per D-04):
    - Regular Supabase Auth user with `app_metadata.is_super_admin = true`
    - Checked at root domain /super-admin/* routes
    - Cross-tenant: no store_id requirement
    - resolveAuth() checks super-admin before staff/customer (per STATE.md decision)

    **## Multi-Tenant Request Lifecycle** (per D-05):
    Full end-to-end Mermaid flowchart diagram showing the complete request lifecycle from 19-RESEARCH.md lines 430-449:
    Request arrives -> webhook passthrough check -> root domain check (super-admin or other) -> subdomain extraction -> tenantCache (hit/miss) -> store active check -> inject x-store-id header -> route type (admin/pos/storefront) -> auth check -> security headers -> response

    Text walkthrough of each step:
    1. Webhook routes (`/api/webhooks/*`) skip all auth, get security headers only
    2. Root domain requests: super-admin gets is_super_admin check, others get session refresh
    3. Subdomain detected: extract slug from host (e.g. "mystore" from mystore.lvh.me:3000)
    4. tenantCache.getCachedStoreId(slug): in-memory Map, 5-min TTL, per-serverless-instance
    5. Cache miss: query stores table, setCachedStoreId. Unknown slug: 404. Suspended: /suspended
    6. Inject x-store-id and x-store-slug headers into request
    7. Route-level auth: /admin requires owner, /pos requires staff JWT or owner, storefront is open
    8. Security headers (CSP report-only, X-Frame-Options, etc.) added to all responses

    Key files: `src/middleware.ts`, `src/lib/tenantCache.ts`, `src/lib/supabase/middleware.ts`

    **## Tenant Isolation Model**:
    Five layers documented:
    1. Subdomain -> store_id (middleware)
    2. tenantCache (in-memory, 5-min TTL)
    3. Custom JWT claims (app_metadata.store_id, set by auth hook in migration 003)
    4. RLS policies (all tables filter by store_id)
    5. SECURITY DEFINER RPCs (complete_pos_sale, complete_online_sale, increment_promo_uses, restore_stock, check_rate_limit) restricted to service_role via REVOKE/GRANT (migration 021)

    **## Feature Gating** (per D-06):
    Read `src/lib/requireFeature.ts` and `src/config/addons.ts` before writing.
    - Three add-ons: xero, email_notifications, custom_domain
    - Fast path: read from user.app_metadata[feature] (JWT claim, no DB round-trip)
    - DB fallback: query store_plans table when requireDbCheck: true (critical mutations)
    - Never throws: returns { authorized: true } or { authorized: false, feature, upgradeUrl }
    - upgradeUrl pattern: /admin/billing?upgrade={feature}
    - Mermaid flowchart: requireFeature(feature) -> check JWT claim -> if missing, check DB -> return { authorized, upgradeUrl }

    **## Billing & Subscriptions** (per D-06):
    Read `src/app/api/webhooks/stripe/billing/route.ts` before writing.
    - Per-add-on subscription model (not plan tiers — no upgrade cliffs, per STATE.md decision)
    - Separate webhook endpoint: /api/webhooks/stripe/billing with STRIPE_BILLING_WEBHOOK_SECRET
    - Subscription events update store_plans boolean columns (has_xero, has_email_notifications, has_custom_domain)
    - Stripe Checkout for subscription creation via createSubscriptionCheckoutSession action
    - Billing Portal for self-service management via createBillingPortalSession action
    - Mermaid sequence diagram: Owner -> createSubscriptionCheckoutSession -> Stripe Checkout -> Webhook -> update store_plans -> feature unlocked

    **## Data Flow: POS Sale**
    Brief end-to-end: Staff scans/selects items -> cart state (React) -> completeSale action -> Zod validation -> resolveAuth (staff JWT) -> complete_pos_sale RPC (SECURITY DEFINER) -> stock decrement + order creation in single transaction -> { success, order }

    **## Data Flow: Online Order**
    Brief end-to-end: Customer adds to cart -> createCheckoutSession -> Stripe Checkout hosted page -> Stripe webhook -> complete_online_sale RPC -> stock decrement + order creation -> email notification (if add-on enabled)
