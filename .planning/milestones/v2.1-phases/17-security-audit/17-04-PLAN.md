  <action>
    Add IP-level rate limiting to the staff PIN login action (SEC-13, Finding Group J).

    **Current state:** `staffPin.ts` has DB-level lockout (10 attempts per staff account in 5-min window) but no IP-level limiting. An attacker could try one PIN per staff ID from the same IP without triggering the per-account lockout.

    **Implementation:**
    Read the `check_rate_limit` RPC from migration 009 to understand its interface. It should accept an identifier and window parameters.

    Add an IP-based rate limit check BEFORE the existing PIN verification logic:

    ```typescript
    // IP-level rate limiting (SEC-13)
    // Get client IP from headers (Next.js provides x-forwarded-for in serverless)
    const headersList = await headers()
    const clientIp = headersList.get('x-forwarded-for')?.split(',')[0]?.trim() || 'unknown'

    // Check IP rate limit using existing check_rate_limit RPC
    const { data: rateLimitResult, error: rateLimitError } = await supabase
      .rpc('check_rate_limit', {
        p_identifier: `pin_login:${clientIp}`,
        p_window_seconds: 300,  // 5 minutes
        p_max_attempts: 20,     // 20 attempts per IP per 5 min (allows multiple staff on same device)
      })

    if (rateLimitError || !rateLimitResult) {
      return { error: 'Too many login attempts. Please try again later.' }
    }
    ```

    The threshold is 20 per IP (higher than the 10 per-account limit) because a legitimate POS iPad may have multiple staff members logging in during shift changes. The per-account lockout (existing 10 attempts) is the primary defense; IP-level limiting is the secondary defense against distributed account enumeration.

    Also verify the existing per-account lockout logic is correct:
    - 10 attempts max in 5-minute window
    - `pin_locked_until` timestamp is set on lockout
    - JWT has 8h expiry, httpOnly cookie, secure in production, sameSite: lax

    Note: If the `check_rate_limit` RPC interface differs from the above (different parameter names, return type), adjust the call accordingly based on what you read in migration 009. The key behavior is: IP-based limiting with 20 attempts per 5-minute window.
  </action>

  </verify>
  <done>Staff PIN login has both per-account DB lockout (existing, 10 attempts/5min) AND IP-level rate limiting (new, 20 attempts/5min via check_rate_limit RPC). An attacker cannot enumerate PINs across staff accounts from a single IP.</done>
