    **Create `src/components/admin/orders/RefundItemSelector.tsx`:**

    `'use client'` component. Props:
    ```typescript
    interface OrderItemForRefund {
      id: string
      product_name: string
      quantity: number
      unit_price_cents: number
      discount_cents: number
      line_total_cents: number
    }

    interface RefundItemSelection {
      orderItemId: string
      quantityToRefund: number
      refundCents: number
    }

    interface RefundItemSelectorProps {
      items: OrderItemForRefund[]
      alreadyRefundedQty: Map<string, number>  // order_item_id -> total qty already refunded
      selections: RefundItemSelection[]
      onSelectionsChange: (selections: RefundItemSelection[]) => void
    }
    ```

    Render a list of order items, each with:
    - Checkbox to include/exclude from refund. Disabled + greyed out (opacity-40) if `alreadyRefundedQty.get(item.id) >= item.quantity` (fully refunded already).
    - Product name label
    - Quantity spinner: min 1, max = `item.quantity - (alreadyRefundedQty.get(item.id) ?? 0)`. Use `<button>` for -/+ with the same 36px visible / 44px touch target pattern from QuantityControl. Display current qty in a `<span>` between buttons.
    - Per-item refund amount displayed: `formatNZD(Math.floor((quantityToRefund / item.quantity) * item.line_total_cents))`
    - Already-refunded indicator: if partially refunded, show "(X of Y already refunded)" in text-text-muted text-xs below the product name

    "Select All" toggle at the top (per D-04): a checkbox labeled "Select All Items". When checked, selects all un-fully-refunded items at their max refundable quantity. When unchecked, clears all selections.

    Total refund amount display at bottom: `formatNZD(totalRefundCents)` in font-bold, calculated as sum of all selected items' refund cents.

    Styling: follow DESIGN.md — DM Sans for text, font-mono tabular-nums for amounts, border-border for dividers, bg-surface for header row.

    **Create `src/components/admin/orders/PartialRefundFlow.tsx`:**

    `'use client'` component. Multi-step flow rendered inside the drawer (same layout pattern as RefundConfirmationStep).

    Props:
    ```typescript
    interface PartialRefundFlowProps {
      order: {
        id: string
        order_items: OrderItemForRefund[]
        payment_method: string | null
        channel: string
        total_cents: number
      }
      existingRefunds: Array<{
        refund_items: Array<{ order_item_id: string; quantity_refunded: number }>
      }>
      onBack: () => void
      onRefundComplete: () => void
    }
    ```

    Three steps managed by `useState<'select' | 'confirm' | 'eftpos_confirm'>('select')`:

    **Step 'select':**
    - Header: "Refund Items" (font-display font-bold text-xl, matching RefundConfirmationStep header)
    - Body: `<RefundItemSelector>` with computed `alreadyRefundedQty` Map from `existingRefunds`
    - Footer: "Continue" button (amber, same style as RefundConfirmationStep confirm button). Disabled when no items selected (`selections.length === 0`).
    - "Back to Order" button below (same border style as RefundConfirmationStep back button).

    **Step 'confirm':**
    - Header: "Confirm Refund"
    - Body:
      - Summary of selected items (product name, qty, amount per item)
      - Total: `formatNZD(totalRefundCents)` in font-bold
      - Payment method message (per D-12):
        - If `order.channel === 'online'` or `order.payment_method === 'stripe'`: "Refund will be processed to the customer's card"
        - If `order.payment_method === 'cash'`: "Hand {formatNZD(totalRefundCents)} cash to the customer"
        - If `order.payment_method === 'eftpos'`: "Process {formatNZD(totalRefundCents)} refund on the EFTPOS terminal"
      - Reason dropdown: same `<select>` as RefundConfirmationStep (customer_request, damaged, wrong_item, other)
      - "This cannot be undone." warning text
    - Footer:
      - If payment_method is 'eftpos': "Continue to Terminal" button (amber). On click -> set step to 'eftpos_confirm'.
      - Otherwise: "Confirm Refund" button (amber). On click -> call `processPartialRefund({ orderId: order.id, reason, items: selections.map(s => ({ orderItemId: s.orderItemId, quantityToRefund: s.quantityToRefund })) })`. On success -> `onRefundComplete()`. On error -> display inline error.
      - "Back" button to go back to 'select' step.
    - `isLoading` state disables both buttons during processing (prevents double-submit per Pitfall 1).

    **Step 'eftpos_confirm' (per D-13):**
    - Full-screen navy overlay matching `EftposConfirmScreen.tsx` exactly:
      - `fixed inset-0 z-[70] bg-navy flex flex-col items-center justify-center px-4`
      - `role="alertdialog" aria-modal="true"`
      - Amount display: `text-sm text-white/60` "Refund NZD" label, then `text-3xl font-display font-bold text-white tabular-nums` amount
      - Heading: `text-xl font-bold text-white mt-8 text-center` "Did the EFTPOS terminal approve the refund?"
      - Subtext: `text-sm text-white/60 mt-2 text-center` "Check the terminal screen before confirming."
      - Two buttons in `flex gap-4 w-full max-w-lg mt-12`:
        - "YES -- Refund Complete" (bg-success, flex-1, min-h-[56px]): calls `processPartialRefund` then `onRefundComplete()`
        - "NO -- Cancel Refund" (bg-error, flex-1, min-h-[56px]): goes back to 'confirm' step
    - Focus trap on mount (copy pattern from EftposConfirmScreen)
    - `isProcessing` state disables buttons while server action runs
